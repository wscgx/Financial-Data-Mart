# 项目搭建问题总结与反思报告

## 一、问题汇总

### 问题1：SQLite字段类型限制（已解决）
- **现象**：SQLite不支持常见的字段类型，只能用REAL和TEXT代替
- **影响**：数据类型精度丢失，约束能力弱
- **解决**：已迁移到MySQL，使用标准数据类型（DECIMAL、VARCHAR、DATETIME等），SQLite遗留代码和financial_dw.db已全部清理

### 问题2：ETL脚本单一SQL实现
- **现象**：Agent生成的ETL都由一段SQL实现
- **影响**：项目扩张、数据量变大时会出现数据倾斜、脚本计算效率过低等问题
- **改进**：采用分层ETL设计（ODS→DWD→DWS→ADS），复杂逻辑拆分到不同层级

### 问题3：中文乱码问题（已解决）
- **现象**：PowerShell执行sqlite3时将UTF-8转为GBK存入数据库
- **解决**：已统一使用MySQL数据库，UTF-8MB4编码，SQLite已清除

### 问题4：前端字段显示英文
- **现象**：生成的前端页面默认使用字段名（英文），而非中文
- **解决**：在AGENTS.md中增加前端展示规范，强制要求前端字段必须使用中文，来源为数据库字段COMMENT注释

### 问题5：数据未落地与注释缺失（已解决）
- **现象**：
  - Agent一次性实现业务需求时，只创建建表语句，不将数据落地到MySQL
  - 中文注释因SQLite原因只给了字符串注释，未定义COMMENT
  - 数据质量校验只做检测，未落地，无法查看测试结果
- **解决**：
  - 制定数据落地规范，要求DDL执行后验证表结构，ETL执行后验证数据
  - 强制使用MySQL原生COMMENT语法（SQLite遗留已清理）
  - 校验结果保存为CSV文件到指定目录

### 问题6：缺乏Mapping文档与规范
- **现象**：
  - 没有构建Mapping文档
  - 需增加Agent开发规范
  - 新增文档没有按类别分好
- **解决**：
  - 建立完整的Mapping文档规范（MD概览+CSV明细）
  - 按需求选项分文件夹存放（option_a/、option_b/、option_e/）
  - 制定AGENTS.md开发规范文档

### 问题7：表结构缺少数据源信息
- **现象**：Agent自动生成的表，表结构没有数据源信息
- **解决**：强制要求所有DWD/DWS/ADS层表必须包含`dm_src_info`字段

### 问题8：ETL脚本SQLite与MySQL语法不兼容（已解决）
- **现象**：Agent生成的ETL脚本使用SQLite语法（INSERT OR REPLACE、DATETIME('now')、julianday等），无法直接在MySQL执行
- **解决**：
  - 重写ETL脚本为MySQL语法（INSERT IGNORE、NOW()、DATEDIFF等）
  - 编写Python加载脚本自动转换语法
  - SQLite遗留脚本已全部删除

### 问题9：多表JOIN时字符集排序规则冲突
- **现象**：MySQL中多表JOIN时出现"Collation mismatch"错误
- **解决**：在JOIN条件中使用`COLLATE utf8mb4_unicode_ci`统一排序规则

### 问题10：复杂窗口函数和日期计算语法差异
- **现象**：SQLite的窗口函数和日期计算语法与MySQL不同（如date()函数、窗口函数支持程度）
- **解决**：使用MySQL原生函数替换（DATE_SUB、DATE_ADD、ROW_NUMBER等）

## 二、问题根因分析

### 1. 技术选型过早固化（已解决）
- **根因**：项目初期选择SQLite作为数据库，未充分考虑数据规模、类型支持和团队协作需求
- **教训**：技术选型应基于业务预期规模、团队技术栈、运维能力综合评估
- **验证**：已统一迁移到MySQL，SQLite遗留代码和db文件已全部清理

### 2. Agent能力边界认知不足
- **根因**：过度依赖Agent自动生成代码，未对其输出进行约束和规范
- **教训**：Agent是辅助工具，需要人类提供：
  - 明确的开发规范（AGENTS.md）
  - 输出格式约束（DDL、ETL、Mapping文档模板）
  - 质量检查机制（数据质量校验规则）
- **改进**：建立Master Agent + 专业子Agent协作体系

### 3. 缺乏架构设计前置
- **根因**：直接让Agent开始生成代码，没有先设计数据流架构、ETL分层、文档管理体系
- **教训**：从0搭建项目必须先有架构设计，再进入编码阶段
- **改进**：制定标准流程：需求→选型→架构→规范→环境→原型→迭代→上线

### 4. 编码与环境问题未预判（已解决）
- **根因**：Windows + PowerShell + SQLite的字符编码组合问题未提前调研
- **教训**：跨平台开发需提前验证编码、路径、工具链兼容性
- **解决**：已统一使用MySQL + UTF-8MB4编码，Python脚本处理中文数据，SQLite已清除

### 5. 数据治理意识薄弱
- **根因**：只关注"能跑通"，忽略了数据血缘、质量校验持久化、字段映射文档
- **教训**：数据项目必须从第一天就建立数据治理规范
- **改进**：建立24+条数据质量校验规则，完整Mapping文档体系

### 6. SQL方言差异未充分考虑（已解决）
- **根因**：Agent默认生成SQLite语法，未考虑目标数据库是MySQL
- **教训**：必须在规范中明确目标数据库，并提供语法转换指南
- **改进**：AGENTS.md中强制要求使用MySQL语法，提供数据类型映射表，SQLite已统一清除

## 六、新增问题与发现（Phase 3 阶段）

### 问题18：数据质量校验发现异常数据（当前）

- **现象**：ETL端到端跑通后，运行39项质量校验，3项未通过
- **详情**：
  1. `dwd_invalid_date`：5,077条异常 — `stat_date`超过当前日期（数据生成时日期范围过宽）
  2. `missing_customer_aum`：375条异常 — 部分客户在`ods_customer`中存在但无AUM数据
  3. `multi_aum_consistency`：1条异常 — 多维分析与平台汇总AUM存在偏差
- **根因**：数据生成脚本(`generate_2months_data.py`)生成的日期范围包含未来日期，以及部分客户未发生交易导致无资产数据
- **教训**：数据生成脚本也需要纳入质量管控，不能只校验ETL产出

### 问题19：Spark容器端口映射调整

- **现象**：迁移计划中Spark UI端口为8083，实际docker-compose中映射为18080→8080
- **影响**：文档与实际不一致，需统一
- **解决**：docker-compose中实际使用18080，文档已同步更新

### 问题20：MySQL端口冲突

- **现象**：本地MySQL已占用3306端口，Docker MySQL映射为3307→3306
- **影响**：ETL脚本中`MYSQL_PORT=3306`连接本地MySQL，Docker MySQL需使用3307
- **教训**：多环境（本地+Docker）共存时，端口规划必须提前做

### 问题21：数据新鲜度滞后

- **现象**：`dwd_customer_asset_daily`最新数据为2026-06-24，非实时
- **根因**：批处理ETL每日执行一次，非实时链路尚未打通
- **后续**：需要完成Phase 3（实时流处理）才能实现分钟级数据新鲜度

### 问题22：ETL调度尚未集成Airflow

- **现象**：MySQL ETL通过`run_all_etl.py`手动执行，未接入Airflow自动调度
- **影响**：生产环境需要人工干预，无法自动化
- **后续**：需配置Airflow DAG，将ETL脚本纳入调度

## 八、从0搭建项目的标准流程

```
1. 需求调研 → 2. 技术选型 → 3. 架构设计 → 4. 规范制定 
→ 5. 环境搭建 → 6. 原型验证 → 7. 迭代开发 → 8. 测试上线
```

每个阶段都必须有明确的输出物和评审机制，不能跳过任何一步直接进入编码。

### 各阶段输出物清单

| 阶段 | 输出物 | 负责人 |
|------|--------|--------|
| 需求调研 | 业务需求文档、指标清单 | 业务分析师 |
| 技术选型 | 技术选型报告、环境配置 | 架构师 |
| 架构设计 | 数据分层设计、数据流图 | 数据架构师 |
| 规范制定 | AGENTS.md、命名规范、文档模板 | 技术负责人 |
| 环境搭建 | 数据库初始化、工具链配置 | 开发工程师 |
| 原型验证 | 原型数据、验证报告 | 开发工程师 |
| 迭代开发 | DDL、ETL脚本、Mapping文档 | ETL开发工程师 |
| 测试上线 | 质量校验报告、上线清单 | 数据质量工程师 |

## 九、总结

本次项目搭建过程中遇到的问题，本质上是**缺乏系统化项目规划**和**对AI工具能力边界认知不足**导致的。从0开始搭建项目，尤其是数据类项目，需要：

1. **先设计，后编码**：架构设计、规范制定必须前置
2. **技术选型要慎重**：考虑当前需求，更要考虑未来扩展
3. **AI是助手不是主角**：需要人类提供约束、规范和审核
4. **数据治理从第一天开始**：文档、血缘、质量校验缺一不可
5. **环境验证不能省**：编码、权限、工具链问题提前暴露
6. **SQL方言要明确**：提前确定目标数据库，避免后期大量转换工作
7. **前端展示要规范**：业务界面100%中文化，提升用户体验
8. **架构演进要渐进**：保留MySQL查询层，Delta Lake做存储计算层
9. **质量校验要贯穿**：ETL跑通不等于数据正确，必须有质量校验闭环
