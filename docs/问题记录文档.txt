1、【已解决】刚开始用agent智能体开发项目的时候，sqlite不支持一些常见的字段类型，只能用real和text代替。已统一迁移到MySQL。
2、agent智能生成的etl目前都是由一段sql实现，在项目小，数据少的时候不会有问题，但是在项目扩张，数据量变大，就会出现类似数据倾斜，脚本计算效率过低等问题。
3、【已解决】中文乱码，PowerShell 执行 sqlite3 时将 UTF-8 转为 GBK 存入数据库。已改为mysql数据库。
4、生成的前端页面默认使用的是字段名，即英文，需要赋予他技能让他将落于前端的字段都改为中文或者其中文注释
5、【已解决】想要智能体能一次性将业务需求全部实现时，发现其只会创建建表语句，不会将数据落地到mysql数据库中，给他新增了这个规则，中文注释也是因为之前是sqlite的缘故导致只给了字符串注释，并没有定义comment。已统一MySQL COMMENT语法。
数据质量校验也是只做检测，并没有落地，无法查看其测试结果，判断是否正确。
6、没有构建mapping文档， 需增加agent开发规范，增加的文档也没有按类别分好，需要分类
7、智能体自动生成的表，表结构没有数据源，需要补充源表主表是哪个
8、在前端bi开发设计的模型中，有些是不符合实际应用需求的，需要认为判断，后续给agent指定规则，哪些内容适合用什么看板
9、dws层应用了ods层的数据，没有做数据隔离，为什么要这么设计

(1. 数据质量无法保证：ODS是原始数据，可能有脏数据、重复、缺失
2. 口径不一致：不同DWS表各自清洗同一份ODS，容易出现逻辑重复或不一致
3. 可追溯性差：问题排查时没有中间清洗层作为锚点
4. 规模扩大后重构成本高：从"扁平"到"分层"的改造代价很大
1. 数据集市模式：很多银行/券商直接建"数据集市"（类似DWS），从ODS直取数据
2. 监管报送驱动：监管指标口径固定，不需要DWD做通用清洗
3. 核心系统数据质量高：银行核心系统（如信贷、理财）本身数据质量就有保障
4. 历史包袱：很多老系统架构就是ODS → 数据集市/报表层)
